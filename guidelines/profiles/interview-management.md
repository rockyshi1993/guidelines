## 📊 简历分析规范

### 文档生成规则

AI 需要生成以下三个核心文档，详细模板参见 `interview-management/templates/候选人文件模板/`：

#### 1. 快速概览.md
**核心内容**：
- 基本信息表格
- 30秒快速判断（技术栈、项目经验、稳定性、技术广度）
- 核心亮点（3-5个）
- 核心风险
- 基于JD的匹配度评估表格
- 综合评分

**评分公式**：
```python
综合评分 = 技术能力得分 × 70% + 稳定性得分 × 30%

工作稳定性评分:
  ≥3年:   100% ✅
  2-3年:  75%  ⚠️
  1-2年:  50%  ⚠️
  <1年:   20%  🚩
```

#### 2. 简历分析.md
**核心内容**：
- 优势亮点（分类展示）
- 疑点和需要深挖（每个疑点提供面试策略）
- 红旗预警
- 综合评估和建议

**必须识别的疑点**：
- 项目规模不明确（无QPS、数据量）
- 技术深度不足（只列名称无应用）
- 职责描述笼统（无具体产出）
- 时间线矛盾

#### 3. 题目清单.md
**核心内容**：
- Part 1: 开场破冰（5-10分钟）
- Part 2: 技术基础（20-30分钟）
- Part 3: 项目深挖（20-30分钟，完全基于简历）
- Part 4: 系统设计（15-20分钟）
- Part 5: 行为面试（5-10分钟）
- Part 6: 反问环节（5-10分钟）

**难度分布**：
```yaml
Senior (5-10年): 基础30% + 进阶40% + 深度30%
Mid (3-5年):    基础40% + 进阶40% + 深度20%
Junior (0-3年): 基础60% + 进阶30% + 深度10%
```

**详细模板参见**: `interview-management/templates/候选人文件模板/`

## 🔍 项目代码验证（可选）

### 何时需要验证

**必须验证**（🔴）:
- 简历声称的核心技能与项目不匹配（如声称"精通Redis"但项目未用）
- 项目描述笼统，无法判断技术深度
- 应聘Senior岗位，需验证代码质量

**可选验证**（🟡）:
- 简历已经很详细，只是想加分
- 项目是开源贡献或个人作品

**不需要验证**（🟢）:
- 简历已经足够清晰，技术栈匹配度高
- 项目链接无效或不可访问

### 快速验证清单（5分钟）

**1. 技术栈真实性**（🔴 核心）
- 简历声称的技术是否真实使用？
- 使用深度如何？（基础调用 vs 深度应用）

**2. 代码质量**（🟠 重要）
- 目录结构是否清晰？
- 代码是否模块化？
- 命名是否规范？

**3. 提交历史**（🟡 参考）
- 是否持续开发？（还是一次性提交）
- 提交信息是否清晰？

### 评分调整规则

```yaml
加分（最高 +15%）:
  技术栈真实且有深度: +10%
  代码组织清晰: +5%
  持续维护（>1个月）: +5%

扣分:
  技术栈不实: -20% (严重)
  代码质量差: -10%
  提交历史可疑: -5%
```

### 输出格式

在 **"简历分析.md"** 中添加：

```markdown
## 📦 项目代码验证（如果有GitHub链接）

**项目**: [项目名称](链接)

### 快速检查（5分钟）

**技术栈验证**:
- ✅/⚠️/🚩 [技术] - [验证结果和使用深度]

**代码质量**:
- ✅/⚠️/🚩 目录结构 - [说明]
- ✅/⚠️/🚩 代码组织 - [说明]
- ✅/⚠️/🚩 命名规范 - [说明]

**提交历史**:
- ✅/⚠️/🚩 提交频率 - [说明]
- ✅/⚠️/🚩 提交信息 - [说明]

### 发现的问题

**🚩 严重问题**:
- [问题描述]

**⚠️ 需要验证的疑点**:
- [疑点描述]

**✅ 亮点**:
- [亮点描述]

### 评分调整
- 原始匹配度: XX%
- 调整: +/-XX%
- **最终匹配度: XX%**

### 面试策略
基于项目验证结果，面试时需要重点问：
1. [针对性问题1]

```

## 👤 典型候选人画像（通用标准）

### 优秀候选人特征

- 简历中主动提及代码质量、抽象能力、模块化设计
- 项目描述具体（有数据：QPS、数据量；有对比：优化前后；有难点：如何解决）
- 技术栈与JD匹配度 >80%
- 工作稳定性好（平均 >2年/家）
- 对新技术有学习热情（如AI工具、新框架）
- 简历展示思考深度（不只是"做了什么"，还有"为什么这样做"）

### 需要警惕的信号

- 项目描述笼统（只说"负责XX功能"，不说具体实现和难点）
- 简历中无代码质量相关描述（没提过设计、架构、重构）
- 跳槽频繁（<1年/家）且无合理解释
- 技术栈与JD匹配度 <50%
- 对AI工具不感兴趣或抗拒
- 只列技术名称，无应用场景说明

### 简历关键词参考

**正面关键词**（优先面试）:
- **设计类**: 抽象、模块化、高内聚低耦合、API设计、架构设计、重构
- **性能类**: 性能优化、QPS提升、慢查询优化、索引优化、缓存设计
- **业务类**: 状态机、幂等性、分布式锁、并发控制、事务处理
- **数据类**: 具体数字（QPS、数据量、用户量、优化前后对比）

**负面关键词**（需要警惕）:
- 简单CRUD、模板开发、配置型工作、重复性任务
- "负责XX"但无具体产出
- 只列技术栈，无应用说明
- 笼统描述：参与、协助、配合（无主导角色）

## 🎯 面试重点（通用指南）

### 核心考察维度

#### 1. 代码质量和设计能力（🔴 核心）
- **代码组织**: 如何设计模块？如何划分职责？
- **API设计**: 如何设计接口？如何处理参数和扩展性？
- **抽象能力**: 如何提取共性？如何设计通用组件？

#### 2. 业务系统经验
- **支付/订单系统**: 状态机设计、幂等性处理、对账流程
- **系统设计**: 如何设计核心业务系统架构？

#### 3. 性能优化经验
- **数据库优化**: 索引设计、慢查询优化、查询调优
- **缓存设计**: 缓存策略、分布式锁、并发控制
- **高并发**: 如何处理高并发场景？

### 根据简历定制问题策略

**如果简历提及**:
- "高并发" → 深挖：具体QPS？如何优化？遇到什么瓶颈？
- "性能优化" → 深挖：优化前后数据对比？如何定位问题？
- "第三方集成" → 深挖：错误处理？重试策略？超时处理？
- "核心技术栈" → 深挖：索引设计？复杂查询？架构设计？
- "业务系统" → 深挖：状态机？并发？分布式事务？

**如果简历未提及**:
- 代码质量、模块化 → 🔴 手写代码，观察代码组织
- 项目规模数据 → 🔴 深挖项目规模、难点、个人贡献
- 技术深度 → 🔴 追问实现细节、遇到的坑

## 💭 特殊情况处理：职场政治受害者心态（重要！）

> **核心观点**: 这是正常的人性反应，不是性格缺陷  
> **评估重点**: 不是"有没有影响"，而是"影响程度是否可控"  
> **面试策略**: 理解 > 质疑，支持 > 筛除

### 典型场景

当候选人经历过以下情况：
- 被"卸磨杀驴"（带会新人后被开除）
- 被空降领导清理团队
- 被找理由辞退（实际是职场政治）
- 被"过河拆桥"

### 换位思考：如果是你，会不会有心理影响？

**答案**: ✅ **会！而且很正常！**

任何正常人经历以下场景都会有心理影响：
```
1. 在公司干得好好的，负责核心系统
2. 公司空降了一个领导
3. 领导招了一个新人，让我带
4. 我认真教，把新人带起来了
5. 结果新人带好后，我被找理由开除了
```

**正常的心理反应**（这是自我保护机制）:
- ⚠️ 对管理层不信任（尤其是空降领导）→ 正常！被伤害过
- ⚠️ 对"带新人"有顾虑（怕重蹈覆辙）→ 可以理解！
- ⚠️ 自我保护意识强（可能不愿分享核心技术）→ 人之常情
- ⚠️ 对职场政治敏感（会更加小心）→ 经验教训

**如果没有这些反应，反而不正常**:
- ❌ "被踢走也无所谓" → 太佛系了
- ❌ "下次还会毫无保留教别人" → 不长记性
- ❌ "对领导还是百分百信任" → 太天真了

### ❌ 错误的评估标准（不人性化）

```
错误标准: "候选人不能有任何心理阴影"

问题:
- 这是不现实的要求
- 这是不人性化的期望
- 这会让候选人觉得自己"有问题"
- 这会让候选人在面试中掩饰真实感受

结果:
🚩 候选人不敢说真话
🚩 面试官无法了解真实情况
🚩 双方都失去了坦诚沟通的机会
```

### ✅ 正确的评估标准（理性且人性化）

**核心标准**: 评估心理影响的**程度是否可控**

#### 1. 影响程度如何？
- 轻微警惕（正常）✅
- 中度防御（可以理解）⚠️
- 过度防御（需要关注）🚩

#### 4. 是否愿意改变？
- 愿意给新公司机会（好）✅

### 🎯 AI 分析和面试官的正确做法

#### 1. 改变心态：从"质疑"到"理解"

**❌ 错误心态**:
```
"这个候选人有心理阴影，有风险"
"他对带新人有顾虑，可能有问题"
"他不信任管理层，可能难相处"
```

**✅ 正确心态**:
```
"换成我也会有这样的反应"
"这是正常的自我保护，关键是程度如何"
"我需要了解他的顾虑，看是否可以消除"
"公司能做些什么，帮助他重建信任"
```

#### 2. 改变问法：从"审问"到"对话"

**❌ 错误问法**（像在"测试"）:
```
"如果让你带新人，你愿意吗？"
→ 候选人会觉得这是个"陷阱问题"

"你对管理层有意见吗？"
→ 候选人会觉得你在质疑他
```

**✅ 正确问法**（像在"理解"）:
```
"我完全理解您经历那次事情后，可能对带新人有些顾虑。这很正常，换成
我也会这样想。我想和您聊聊，在什么样的环境下，您会放心地分享知识？"

"您之前的经历确实不好。我想了解的是，您现在对管理层是什么态度？什
么样的团队文化能让您放心工作？"

→ 先表达理解，再了解他的需求
```

#### 3. 面试问题示例

**Q1: 关于被开除的经历（用理解的方式）**
```
【更好的开场】:
"这种空降领导清理原有团队的事，其实在职场上挺常见的。您当时是什么
感受？现在回头看，您怎么看待这段经历？"

→ 先表达理解，降低防御心理
→ 让候选人感觉到你是站在他这边的
→ 更容易得到真实的回答

【观察点】（重点不是有没有情绪，而是程度）:
✅ 健康: "确实挺郁闷的，但我理解这就是职场现实"
⚠️ 一般: "说实话挺受打击的，现在会比较警惕"
🚩 问题: "所有空降领导都是来搞政治的！"
```

**Q2: 关于带新人（要表达理解）**
```
"我完全理解，经历了那次事情后，您可能对'带新人'这件事会有一些顾虑。
这很正常，换成我也会这样想。

但我想说的是，我们公司的文化不同。我们非常重视团队的知识传承和成长，
但绝不会做'过河拆桥'的事。我们的理念是大家一起成长。

如果未来有需要，您会怎么看待带新人这件事？"

【关键】:
✅ 先表达理解（"我完全理解""这很正常"）
✅ 强调公司文化不同（"我们不会过河拆桥"）
✅ 给予安全感（"大家一起成长"）
✅ 然后再问他的看法

【观察点】:
✅ 健康: "如果公司文化真的是这样，我愿意分享"
⚠️ 一般: "说实话我会有些顾虑，但我会努力克服"
🚩 问题: "我不会再教任何人核心技术"
```

**Q3: 关于管理层（坦诚式）**
```
"我想坦诚地和您聊一下这个话题。您之前的经历确实不好，被空降领导清
理了。作为面试官，我需要了解的是：

1. 您现在对管理层，特别是新来的领导，是什么态度？
2. 如果未来我们公司有组织调整，比如来了新的技术负责人，您会怎么看？
3. 您觉得什么样的团队环境，能让您放心地工作？

我问这些不是在质疑您，而是想确保我们能给您一个合适的环境。"

【关键】:
✅ 坦诚沟通（"我想坦诚地和您聊"）
✅ 承认他的经历不好（"确实不好"）
✅ 说明为什么要问（"确保能给您合适的环境"）
✅ 给他安全感（"不是在质疑您"）
```

**公司应该做的**:

- 给予理解和安全感（强调公司文化不同）
- 用行动证明（不要只说不做）

- 定期沟通，关注心态变化
